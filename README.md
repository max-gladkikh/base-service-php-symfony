# Base web service php+symfony

Базовый web service нужен для того, чтобы развернуть готовое окружение для начала разработки.

## Минимальные системные требования:

1 гб оперативки </br>
2 ядра процессора </br>
10гб жесткий диск </br>

Технологии, которые необходимы для развёртки базового сервиса и работы с ним:
1. git
2. docker
3. docker compose

---

## Используемые технологии

Тут описаны те технологии, которые уже присутствуют в этом базовом сервисе, после его развёртки. </br>
Версии этих технологии регулируются в переменных окружения в файле .env </br>
Базовую версию этого файла можно получить скопировав содержимое файла .env_example </br>
Если какая-то из технологий вам не нужна, просто закоментируйте или удалите её из docker-compose.yml

1. nginx
2. php
3. composer - менеджер пакетов php
4. symfony - фреймворк php (опционально, можно не устанавливать в шагах инструкции по установке)
5. postgres 
6. redis

---

## Инструкция по установке

Эта инструкция поможет вам развернуть базовый сервис

1. [Установите git](https://git-scm.com/) (если еще не установлен)
2. [Установите docker](https://docs.docker.com/engine/install/) (если еще не установлен) </br>
   Возможно вместе с докером установится и docker compose </br>
   Проверить это можно командой: </br>
   docker compose version </br>
   Если версия выводится, значит docker compose у вас установлен и шаг 3 можно пропустить
3. [Установите docker compose](https://docs.docker.com/compose/install/) (если еще не установлен)
4. Перейдите в директирию где вы хотите расположить сервис, например:
   cd /home/{USER_NAME}/www
5. Склонируйте данный базовый сервис с помощью инструкции по клонированию из репозитория (желательно клонировать по ssh), например:
   git clone git@github.com:max-gladkikh/base-service-php-symfony.git
6. Переименуйте папку с сервисом на название вашего сервиса
   mv base-service-php-symfony {YOUR_PROJECT_NAME}
7. Зайдите в папку с вашим сервисом
   cd {YOUR_PROJECT_NAME}
8. Копируем базовую конфигурацию переменных окружения
   cp .env_example .env
9. Меняем значение переменной окружения [COMPOSE_PROJECT_NAME](https://docs.docker.com/compose/environment-variables/envvars/) на название вашего нового сервиса.
10. Меняем значение переменной окружения PROJECT_URL на урл сервиса
11. Собираем образы докер контейнеров
    docker compose build --no-cache
12. Проверяем что для первого запуска nginx настроен на 80 порт, чтобы получить первый сертификат ssl (TLS) для https.
    Для этого значение переменной окружения APP_ENV должно быть равно dev и шаблон nginx.conf.template в папке dev настроен на 80 порт.
13. Проверяем что у скрипта дампа базы /postgres/dump_scripts/dump.sh есть права на исполнение, если нет, выдаём ему эти права
    chmod +x dump.sh
14. Запускаем образы докер контейнеров
    docker compose up -d
    Если при запуске выдаёт ошибку с портами (port is already allocated), т е ругается на пересекающиеся порты (с уже запущенными сервисами), то нужно просто сменить в энвах (только outer, те только внешние) порты, которые пересекаются. Если меняли порт у nginx, то у него требуется поставить такой же внутренний порт, как и внешний, который вы задали.
15. Смотрим что все контейнеры запустились командой docker ps -a
    Если какой то контейнер упал при запуске (его статус Exited) то смотрим логи этого контейнера командой docker logs -f --tail 100 DOCKER_CONTAINER_ID и чиним проблему
16. Проверяем что база данных и пользователь успешно создались
    docker logs -f --tail 100 DOCKER_CONTAINER_ID
    Если нет, то смотрим что пишут логи и [доку](https://hub.docker.com/_/postgres#initialization-scripts)
17. Проверяем что дамп базы создаётся, смотрим логи контейнера делающего дампы
    docker logs -f --tail 100 DOCKER_CONTAINER_ID
18. Проверяем есть ли у нас в сервисе файл php.ini для текущей версии пхп, которая указана в .env.
    Файлы php.ini для dev и prod окружения лежат в папке сервиса ./php/ini/\${PHP_PROJECT_VERSION}/php.ini-\${APP_ENV} .
    Если файлов для текущей версии пхп нет, берем их из запущенного контейнера с php из папки /usr/local/etc/php
19. Создаём сертификат для https с помощью letsencrypt
    а) для localhost нужен самоподписанный сертификат. Тут инструкция как его сделать.
    https://letsencrypt.org/ru/docs/certificates-for-localhost/
    // только хром будет ругаться что подключение не безопасно, но нам просто иногда https нужен
    b) Для реального домена нужен сертификат подписанный letsencrypt. Тут инструкция как его сделать.
    https://letsencrypt.org/getting-started/
    Если всё сделано по инструкции с помощью certbot, то будет автоматически добавлено автопродление сертификата
    https://certbot.eff.org/instructions
    В крон crontab -e (/etc/crontab/ or /etc/cron.*/*)
    Или сюда systemctl list-timers
    c) Если нужно сделать/продлить сертификат где нет доступа запустить certbot,
    то на любой машине, хоть на локальном компе выполняем команду
    certbot certonly --manual
    Далее вводим домен и нам по интструкции скажут просто создать файл с нужным содержанием,
    чтобы проверить, что домен нам принадлежит.
    После проверки сертификат, в виде четырёх файлов окажется там где была выполнена команда certbot.
    Основных два, это приватный ключ privkey.pem и публичный ключ fullchain.pem
    Их содержимое грузим туда, где лежит сайт.
20. Кладём оба файла созданного сертификата в дирикторию nginx/ssl
21. Останавливаем и удаляем запущенные контейнеры (где nginx запускался на 80 порт)
    docker compose stop
    docker compose rm
22. Теперь после получения сертификата для https можно настроить nginx на 443 порт,
    например задав переменной окружения APP_ENV значение prod
    или для dev окружения можно просто отредактировать шаблон nginx.conf.template в папке dev на 443 порт и подключить сертификаты по примеру из шаблона nginx.conf.template в папке prod
23. Запускаем образы докер контейнеров
    docker compose up -d
    Если при запуске выдаёт ошибку с портами (port is already allocated), т е ругается на пересекающиеся порты (с уже запущенными сервисами), то нужно просто сменить в энвах (только outer, те только внешние) порты, которые пересекаются. Если меняли порт у nginx, то у него требуется поставить такой же внутренний порт, как и внешний, который вы задали.
24. Смотрим что все контейнеры запустились командой docker ps -a
    Если какой то контейнер упал при запуске (его статус Exited) то смотрим логи этого контейнера командой docker logs -f --tail 100 DOCKER_CONTAINER_ID и чиним проблему

### Установка фреймворка symfony

1. После того как убедились что все контейнеры запущены, заходим в контейнер с php
   docker exec -it PHP_CONTAINER_ID bash
2. [Устанавливаем symfony cli](https://symfony.com/download) примерно вот такой командой:
   curl -sS https://get.symfony.com/cli/installer | bash && mv /root/.symfony5/bin/symfony /usr/local/bin/symfony && mkdir /root/.config && mv /root/.symfony5 /root/.config/symfony-cli
3. Проверяем что symfony cli успешно установился командой
   symfony check:requirements
4. (Этот шаг опциональный) Для того чтобы зафиксировать внутри сохранённый symfony cli можно выйти из контейнера php и сохранить его состояние в новый образ командой:
   docker commit DOCKER_CONTAINER_ID
   Закоммитив контейнер вы увидите новый образ в списке всех образов docker images, но он будет безымянный. Зададим образу имя и тег командой:
   docker tag DOCKER_IMAGE_ID DOCKER_IMAGE_NAME:DOCKER_IMAGE_TAG
5. Внутри контейнера с php, заходим в папку которая пробрасывается наружу
   cd /home/\${COMPOSE_PROJECT_NAME}/www
   удаляем файл .gitkeep если он там есть
   и [разворачиваем в ней фреймворк symfony](https://symfony.com/doc/current/setup.html).
   Для того чтобы развернуть фреймворк в текущей директории, а не в новой, вместо названия папки нужно указать точку, например так (symfony new . --version="X.X.*" --webapp)
6. Удаляем папку /www/.git
7. Выходим из контейнера php командой
   exit
8. Меняем права на все вложенные файлы и папки внутри директории www командой:
   sudo chown -R OWNER:GROUP www/
   тоже самое делаем с папкой .ssh/ :
   sudo chown -R OWNER:GROUP .ssh/
   Ставить можно например те права, которые у вас установлены на других файлах и папках вашего сервиса, т е например на папке /nginx или на файле /README.md эти права пользователя и группы должны быть одинаковые.
9. Переопределяем путь к .env файлу в композере.
   Для этого в файле composer.json находим раздел extra
   и задаём/переопределяем параметр runtime.dotenv_path на значение "../.env".
   Пример как это сделать есть в [инструкции](https://symfony.com/doc/current/configuration/override_dir_structure.html).
   После этого нужно применить изменения запустив внутри контейнера с PHP команду composer dump-autoload
10. Переопределяем путь к .env.test файлу для тестов.
    Для этого в файле www/tests/bootstrap.php меняем путь к .env файлу на dirname(\_\_DIR\_\_).'/../.env.test'
11. Проверяем что сервис заработал:
    - (Nginx + php) Если в .env у вас указано значение переменной окружения NGINX_HOST по умолчанию base.localhost, то перейдите в браузере по адресу base.localhost и вы увидите стартовую страницу symfony. Если вы вместо неё видите ошибки, то логи вам в помощь: логи любого докер контейнера docker logs -f --tail 100 DOCKER_CONTAINER_ID или логи nginx в папке /nginx/log
    - (DB) Подключитесь к БД с помощью вашей IDE, введя параметры, которые вы задали в файле .env, подключение должно быть успешным.
    - (Redis) Подключитесь с помощью вашей IDE
12. Добавляем в composer.json всё что мы хотим добавить, после этого выполняем
    composer update
    composer install --no-scripts --optimize-autoloader
    composer dump-autoload --optimize
    composer check-platform-reqs

### Чистим ненужные файлы и коммитим инициализацию вашего сервиса
1. После развертки приложения symfony, смотрим файлы /www/.env и /www/.env.test, которые создала symfony. Если их содержимое как то дополнит то, что у нас есть в файле .env (это возможно в случае новой версии), то дополняем наши файлы .env и .env_example, после этого удаляем файлы /www/.env и /www/.env.test
2. Так же смотрим файл /www/.gitignore, который создала symfony. Если его содержимое как то дополнит то, что у нас есть в файле .gitignore (это возможно в случае новой версии), то дополняем наш файл .gitignore, после этого удаляем файл /www/.gitignore
3. Так же смотрим файлы www/docker-compose.yml и www/docker-compose.override.yml. Если их содержимое как то дополнит то, что у нас есть в файле docker-compose.yml, то дополняем наш файл docker-compose.yml, после этого удаляем файлы www/docker-compose.yml и www/docker-compose.override.yml
4. Закомитьте инициализацию вашего нового сервиса.
